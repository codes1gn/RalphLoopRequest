Brief summary: kernels/manifest.json structure (CrokTile kernel registry v3)

Top-level keys and meaning:
- version: schema/registry version (3).
- description: human-readable purpose — kernels keyed by (dtype, scenario, shape); region clustering for seed selection; v3 notes 7 scenarios, 10 sweep lines, 134 unique shapes per dtype.

- hardware: GPU environment — model, SM arch, CUDA/driver versions, peak TFLOPS (f16/f8), SM count, SMEM per SM, L2 size.

- choreo_repo, gemm_sp_paper_repo, gemm_sp_kernel_dir: absolute paths linking this project to Choreo and GEMM-SP kernel sources.

- reference_kernels: per-dtype (f16, e4m3) best tuned kernel .cu path, best TFLOPS, hardware efficiency, shape used for that tune, origin iteration, README path.

- baseline_kernels: canonical baseline .co filenames per dtype (starting point for from-scratch tuning).

- scenarios: named benchmark geometries — each has label, type (single vs tied_pair), dimension pattern (M/N/K as int or "S" for swept), sizes arrays; tied_pair scenarios add sub_series with fixed dims and tied_sizes or vary_sizes.

- size_constraints: minimum M, N, K allowed for shapes.

- region_clustering: near/far thresholds relative to reference shape (4096,8192,8192); typical_shapes_for_scratch lists representative far (M,N,K) per dtype for full scratch tuning (_fs artifact suffix).

- build: nvcc flags/includes/libs, dtype-specific extras, choreo_flags, warmup and iteration counts for benchmarking.

- verification: per-dtype numeric tolerances (base_tol, rel_tol) for correctness checks.

- tuning_limits: max iterations and descriptions for from_current_best vs from_scratch modes.
