Code review: harness/checkpoint_harness.py
==========================================

Scope
-----
Single-file CLI that (1) simulates an AskQuestion-style checkpoint with preset answers, (2) verifies transcripts against a regex-shaped “checkpoint block,” and (3) runs a bundled test suite. Review covers design, correctness, and improvements.

Design
------
Strengths
- Clear separation of concerns: `simulate_askquestion` (simulation), `verify_transcript` (parsing/validation), thin `cmd_*` handlers, and `main` for argparse wiring.
- `CheckpointResult` as a dataclass keeps structured output for JSON and batch reporting; `asdict` is appropriate for serialization.
- Docstring at top documents subcommands and typical usage; argparse subparsers mirror that structure.
- `TASK_CONTEXTS` centralizes scenario data for the suite without scattering literals.

Weaknesses / coupling
- `cmd_test_suite` hard-codes slice `all_results[5:15]` for “consecutive” results. That assumes exactly five entries from Group 1 (`len(TASK_CONTEXTS)`). Adding or removing a context silently misaligns the slice and misreports pass rate. Prefer tagging results (e.g. `phase="consecutive"`) or slicing by `test_id` range, or append Group 2 to a dedicated list.
- Transcript verification is regex-first; the “spec” is implicit in the pattern and duplicated in the sample `good_transcript`. A small schema (or shared fixture file) would reduce drift between tests and production expectations.
- `verify_transcript` mixes boolean flags and a synthetic `"pass"` aggregate; callers must know the full contract. A typed result (dataclass or TypedDict) would clarify the API.

Correctness
------------
Simulation (`simulate_askquestion`)
- Empty options: returns `success=False` with an error message. Correct for “must choose.”
- Matching: accepts substring match of normalized response in each option’s lowercased text, or numeric string `"1"`..`n`. Correct for typical CLI use; substring matching can admit false positives (e.g. response `"it"` matching `"Iterate implementation"` is unlikely but `"2"` vs twelve options is unambiguous).
- `options_contextual`: `has_context and has_done` with `has_context = len(options) >= 3`. This encodes a product rule (≥3 options and a Done-like label). Consistent with `verify_transcript`’s `num_options >= 3 and has_done_option`. Document the definition so it stays aligned with upstream “contextual” semantics.

Verification (`verify_transcript`)
- `CHECKPOINT_PATTERN` enforces a specific layout: `**Completed:**` line, blank line, `**What's next?**`, one or more `N. ...` lines, optional newline, then `Or just tell me what to do next:`. Valid good transcript in the suite satisfies this.
- Edge case: if the checkpoint block matches but `**Completed:**` appears only inside the block and not elsewhere, `has_completed_line` still becomes true via the separate `if "**Completed:**" in text` check—reasonable.
- Using the key name `"pass"` in a dict is legal but shadows the `pass` statement mentally; `"passed"` would read clearer.

Test suite (`cmd_test_suite`)
- Group 1 length and Group 2 slice: mathematically consistent for current `TASK_CONTEXTS` (5 + 10). Fragile as noted under design.
- Edge-case expectations: empty options should fail (`not r.success` → PASS), single “Done” should succeed, mismatched response should fail. Logic matches those prints.
- Summary accounting: `total_all = successes + edge_correct + transcript_correct` mixes checkpoint success counts with hardcoded `edge_correct=3` and `transcript_correct=2`. If an edge case assertion were wrong, the printed “Overall” could still claim full edge/transcript credit. Prefer deriving counts from actual assertions.
- **I/O correctness**: The suite unconditionally opens `/tmp/durable-verify/results/harness-test-suite.json` for writing. There is no `os.makedirs` for that path; on a clean machine this raises `FileNotFoundError` and aborts the suite after printing the summary header work. This is a real robustness bug for the `test-suite` command.

CLI / UX
----------
- `verify` and `verify-stdin` duplicate the same body except input source; could be one function with a string parameter.
- File reads use default encoding; explicit `encoding="utf-8"` improves portability for transcript paths on non-UTF-8-default locales.

Potential improvements
----------------------
1. Create parent directories before writing the JSON report (or write next to the script / to `$TMPDIR` with `tempfile` and print the path).
2. Replace magic slice `[5:15]` with explicit separation of result batches or dynamic indexing from `len(TASK_CONTEXTS)`.
3. Extract `good_transcript` / `bad_transcript` to test data files and optionally reuse snippets in documentation.
4. Consider TypedDict or dataclass for `verify_transcript` return value; use `"passed"` instead of `"pass"`.
5. Add minimal unit tests (pytest) for `verify_transcript` and `simulate_askquestion` so refactors to regex do not rely only on manual suite output.
6. Narrow substring matching (e.g. full-option equality after strip, or word-boundary rules) if false positives become a concern.
7. Return non-zero from `test-suite` if any subgroup fails, for CI use (currently only `cmd_call`, `cmd_verify`, and `cmd_batch` exit with failure codes in obvious paths—check whether `test-suite` should `sys.exit` based on computed failures).

Overall
-------
The harness is readable and fit for purpose for local checkpoint simulation and transcript smoke checks. The main correctness gap is unconditional write to a fixed `/tmp/...` path without ensuring the directory exists, plus fragile indexing in the bundled suite. Addressing those would make the tool reliably runnable in automation and after harmless edits to `TASK_CONTEXTS`.
