Top 3 most important rules from skill/SKILL.md (durable-request)
================================================================

1) Never end silently after completing work (“The One Rule”)
   - At any natural stopping point after doing work (code, files, research,
     debugging, config, etc.), the agent MUST present an interactive checkpoint
     before ending the turn. There are no exceptions except pure informational
     Q&A with no actions taken.
   - Why it matters: the skill’s purpose is to keep the request durable—so the
     user can choose next steps in the same session instead of the agent
     vanishing after output.

2) Anchor the checkpoint with TodoWrite, then use the environment’s tool
   - Before (or batched with) AskQuestion / AskUserQuestion / question, register
     a todo such as durable-checkpoint as in_progress. After the user responds,
     mark it completed.
   - Why it matters: the todo is a structural lock and audit trail—if the
     interactive tool fails or is missing, work cannot be “finished” quietly;
     the agent must resolve the obligation via explicit fallback.

3) Be verbose and explicit about tool outcomes—never silent fallback
   - The agent must state which tool was tried and whether it succeeded, failed,
     or was unavailable (e.g. subagent/CLI), then follow the prescribed fallback
     text and conversational checkpoint when needed.
   - Why it matters: silent success or silent fallback hides failures and
     breaks trust; explicit reporting is marked EXTREMELY-IMPORTANT in the
     skill so users always know why they see a text checkpoint instead of a UI
     widget.
