Code review: harness/checkpoint_harness.py
============================================

Scope: CLI harness that simulates AskQuestion-style checkpoints, verifies transcript shape, and runs batch/test-suite flows.

---

1. Design
---------

Strengths
- Clear separation of concerns: `simulate_askquestion` (behavior), `verify_transcript` (format), thin `cmd_*` handlers, and argparse wiring in `main()`.
- `CheckpointResult` as a dataclass gives stable JSON-serializable output for automation and CI-style checks.
- Docstring at module top explains intent, subcommands, and example invocations; matches how operators would discover the tool.
- `TASK_CONTEXTS` centralizes realistic scenarios for the test suite without scattering literals.

Weaknesses / coupling
- Transcript verification is tightly coupled to one heading string (`**What's next?**`). If product copy changes to another prompt (e.g. only numbered options under `**Completed:**`), `verify_transcript` and `CHECKPOINT_PATTERN` must change in lockstep; consider making the ÔÇťoptions headerÔÇŁ configurable or documenting it as part of the contract.
- `cmd_test_suite` reports ÔÇťconsecutiveÔÇŁ results by slicing `all_results[5:15]`. That assumes exactly five entries in Test Group 1. The logic is correct today but is a silent coupling: adding or removing a `TASK_CONTEXTS` entry breaks the slice without a test failure that points at the cause. Prefer capturing group-2 results in a dedicated list or tagging results with a `phase` field.
- Hard-coded output path `/tmp/durable-verify/results/harness-test-suite.json` ties the harness to a specific environment layout; callers cannot redirect output, and the directory may not exist (see Correctness).

---

2. Correctness
--------------

Matching logic in `simulate_askquestion`
- Response matching uses `normalized_response in o.lower()` for each option. Substring containment can produce false positives (e.g. a short token matching inside a longer label) or ambiguous matches if two options share substrings. For a simulator this may be acceptable; for strict fidelity to ÔÇťpick one option,ÔÇŁ prefer equality after normalization, or longest / unique match rules.
- Numeric selection (`str(i+1)`) is a reasonable extension; good that it is documented implicitly through behavior.

`options_contextual` consistency
- In `simulate_askquestion`, `options_contextual` is `len(options) >= 3` and any option contains `"done"` (case-insensitive).
- In `verify_transcript`, `options_contextual` requires `num_options >= 3` and a ÔÇťDoneÔÇŁ option in the parsed list. The notions align for typical transcripts but differ slightly from `simulate` (which keys off any substring `"done"` in any option, not necessarily a dedicated Done line). Worth unifying the definition in one helper to avoid drift.

Regex verification
- `CHECKPOINT_PATTERN` requires a specific block shape. Valid transcripts that include extra prose before/after can still match via `search`, which is good. Trailing content after `Or just tell me what to do next:` is allowed by the pattern as written; confirm that is intended.
- `OPTION_LINE.findall(match.group(1))` depends on `findall` with a single capture group returning list of strings; correct for extracting option text.

Test suite fragility
- Writing to `/tmp/durable-verify/results/harness-test-suite.json` without `os.makedirs(..., exist_ok=True)` will raise if the directory tree is missing; the suite can crash after printing console output. Either create parents, write next to the script / cwd, or accept `--output` with validation.
- Summary arithmetic (`edge_correct = 3`, `transcript_correct = 2`) duplicates expectations already asserted in prints; if an edge case assertion is wrong, the summary can lie. Derive counts from actual booleans (e.g. `int(empty_failed) + int(...)`) to keep the summary honest.

Exit codes
- `cmd_call`, `cmd_verify`, `cmd_batch` use `sys.exit(0|1)` based on success; consistent and script-friendly.

---

3. Potential improvements
-------------------------

- Add `pathlib.Path` and optional `--output` for the JSON report; default to a temp dir via `tempfile` or the current working directory.
- Replace magic slice `[5:15]` with explicit collection of the 10 loop results, or compute `start = len(TASK_CONTEXTS)` and slice `[start:start+10]`.
- Extract shared ÔÇťcontextual optionsÔÇŁ predicate into a function used by both `simulate_askquestion` and `verify_transcript`.
- Consider `TypedDict` or a small dataclass for `verify_transcript` return type instead of a mutable dict with a `pass` key (valid but easy to mistype); alternatively rename to `"passed"` in the JSON output for clarity.
- Unit tests (pytest) for: empty options, substring collision in matching, regex edge cases (Windows newlines, missing blank line), and directory creation for suite output.
- For CLI ergonomics: `verify` could accept `-` for stdin to avoid a separate `verify-stdin` subcommand, or document both equally in `--help` examples.

---

Overall
-------

The file is readable, small enough to audit, and suitable as a harness for checkpoint-shaped automation. The main risks are environment-specific file output, brittle indexing in the test suite summary, and loose substring matching in the simulator. Addressing those would improve long-term maintainability and trust in automated ÔÇťpassÔÇŁ totals.
