Top 3 most important rules from skill/SKILL.md (durable-request)

1) Never end silently — the “One Rule”
   After ANY task or natural stopping point where work was done, the agent MUST present an interactive checkpoint before ending the turn. The skill applies to code, files, research, debugging, config, etc. The only carve-out is pure informational Q&A with no actions. Explanation: this is the skill’s purpose — making the request “durable” by blocking silent hand-offs so the user can steer the next step in the same request instead of the agent disappearing.

2) Anchor with TodoWrite before the interactive tool
   Before AskQuestion (or equivalent), register an in-progress todo (e.g. id `durable-checkpoint`) so the turn cannot cleanly end with unfinished checkpoint work, and pair TodoWrite with the interactive call in the same batch when possible. Mark it completed after the user responds. Explanation: this creates a structural lock and an audit trail — if the UI tool fails or is missing, the in-progress todo forces an explicit fallback instead of a silent exit.

3) Be explicit about tool attempts and outcomes — no silent fallback
   Always state which interactive tool was tried and whether it succeeded, failed, or was unavailable; if unavailable (e.g. subagent/CLI), say why and use the conversational fallback (summary + numbered “what’s next” options). Never silently skip the checkpoint or silently “succeed” without reporting. Explanation: visibility prevents the anti-pattern the skill targets — the user must always see whether the durable gate ran and how to continue.
