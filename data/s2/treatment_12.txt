Task 12 — JSON config files and schema summary

Scope
-----
The repository contains hundreds of JSON files (kernel bank `meta.json` per shape, duplicated trees under `tuning_v2/`, aggregated results, FSM state, etc.). Below are the **main config categories** and their logical schemas (field sets may vary slightly by file age).

1. `kernels/manifest.json` (registry v3)
-----------------------------------------
- `version`, `description`
- `hardware`: gpu, sm, cuda, driver, peak TFLOPS, n_sm, memory-related caps
- `choreo_repo`, `gemm_sp_paper_repo`, `gemm_sp_kernel_dir` — absolute paths to external repos
- `reference_kernels`: per-dtype `best_cu`, `best_tflops`, `tuned_shape`, `origin_iter`, `origin_readme`
- `baseline_kernels`: f16/e4m3 baseline `.co` filenames
- `scenarios`: nested scenario definitions (single-dim sweeps, tied_pair sub_series, sizes arrays)
- `size_constraints`, `typical_shapes_for_scratch`, region clustering keys, etc.

2. `tuning/state.json`
----------------------
- `version`, `description`
- `shapes`: map of shape keys → objects with `status`, `mode`, `current_iter`, `best_tflops`, `baseline_tflops`, `best_kernel`, `scenarios` (list), `note`, optional `consecutive_discards`, `best_iter`, etc.

3. `tuning/checkpoints/<key>.json`
-----------------------------------
- `key`, `dtype`, `shape` [M,N,K], `mode`, `max_iter`, `current_iter`, `best_iter`, `best_tflops`, `baseline_tflops`, `best_kernel` (path), `consecutive_discards`, `status`, `last_updated`

4. `.claude/skills/fsm-engine/state/loop-state.json` (and variants)
-------------------------------------------------------------------
- `schema_version`
- `fsm`: `current_state`, `iteration`, `max_iteration`, `shape_key`, `dtype`, `shape` [M,N,K], `mode`
- `guard_flags`: booleans for pipeline steps (ncu, compile, verify, git, etc.)
- `metrics`: TFLOPS, decisions, consecutive_discards, bottleneck tags, paths to best kernel
- `paths`: `shape_dir_logs`, `shape_dir_srcs`, `shape_dir_perf`, `checkpoint_file`, `idea_log`
- `completion_promise`, `last_updated`

Related: `loop-state_from_scratch.json`, `push-marginal-state.json`, `iterative-improve-state.json` — same general FSM shape with mode-specific fields.

5. `tuning/push_marginal_queue.json`
------------------------------------
- `description`, `dtype`, `total`, `current_idx`
- `queue`: array of entries with `key`, `M`, `N`, `K`, performance vs `cusparselt`, `status`, `phase1_best`, `phase2_best`, `final_best`, `best_tag`, `note`

6. `tuning/kernel_bank/<dtype>/<MxNxK>/meta.json`
-------------------------------------------------
- `shape`, `M`, `N`, `K`
- `source_type`, `source_tag`, `source_origin`
- `extra_nvcc_flags`
- `prior_tflops`, `cusparselt_tflops`, `measured_tflops`
- `status`, `verified`, `measure_status`
- Optional: `hw_eff_pct` (f16)

7. `tuning/kernel_bank/results_<dtype>.json`
--------------------------------------------
- Dict keyed by shape string → full meta object (flattened registry).

8. `tuning_v3/paper_data.json`, `tuning_v3/all_results.json`
------------------------------------------------------------
- Aggregated experiment outputs for plotting (structure: run metadata + per-shape or per-experiment metrics — use files as source of truth when plotting).

9. `.cursor/` and editor configs
---------------------------------
- May include JSON (e.g. MCP config); not tuning-specific.

10. Package locks
-----------------
- `figures/package-lock.json` — npm dependency lockfile (not application config).

Note: Many generated or copied JSON files under `ralph-loop-request-data/` embed full manifest or log dumps; treat as experimental artifacts rather than hand-maintained schemas.
