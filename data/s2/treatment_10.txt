Task 10 — tuning/ directory structure and workflow

Overview
--------
The `tuning/` tree is the working area for GPU kernel auto-tuning experiments: sweeps, kernel banks, per-shape iteration artifacts, and comparison against cuSPARSELt. It is namespaced by shape key (`<dtype>_<M>x<N>x<K>`) and integrates with `kernels/manifest.json` and external `choreo` builds.

Top-level layout (conceptual)
-----------------------------
- `state.json` — Global registry of every scenario shape: status (e.g. done), mode, iteration counts, best TFLOPS, paths to registered best kernels under `kernels/`, scenario tags, notes.
- `logs/<key>/` — Per-shape iteration logs (e.g. `results.tsv`) and sometimes `idea-log.jsonl` for deduplication.
- `srcs/<key>/` — Kernel source iterations (`seed.cu`, `iterNNN_<tag>.cu`, push-marginal `_pm` variants, etc.). Historical sweeper trees also used `sweeper_srcs/<dtype>_<M>x<N>x<K>/` for per-shape variant sources.
- `perf/<key>/` — Timing text outputs, profiler artifacts, iteration-labeled files.
- `checkpoints/<key>.json` — Crash-safe per-shape checkpoint (current iter, best kernel path, status).
- `kernel_bank/` — Curated per-(dtype,shape) directories with `kernel.cu`, `meta.json`, compiled `kernel` binary, plus `registry_*.tsv`, `results_*.json` aggregates. Subdirs `e4m3/` and `f16/` hold shape-named folders (e.g. `4096x8192x8192`).
- `sweeper_srcs/` — Library of `.cu` variants (f16/e4m3) for automated idea sweeps (macros: L2, nanosleep, maxreg, fences, etc.).
- `best_suite/` — Scripts to compile/measure a suite of “best” kernels (`run_best_suite.py`, `resweep_best.py`).
- `exhaust_logs/` — Exhaustive sweep logs by dtype.
- `perf/` (root under tuning) — Large tree of `perf/<dtype>_<M>x<N>x<K>/` timing folders for many shapes.

Workflow represented
--------------------
1. **Shape schedule** — Shapes come from manifest scenarios (square, sweep_m/n/k, tied-pair sweeps). Dispatcher/skills prioritize by distance from reference shape (4096×8192×8192) and region (near vs far).

2. **Per-shape loop** — For each key: establish baseline → profile (optional/required by mode) → propose change → compile (nvcc + choreo headers) → verify → measure TFLOPS → KEEP/DISCARD → append `results.tsv`, update checkpoint, commit. FSM in `.claude/skills/fsm-engine/` formalizes steps.

3. **Sweeper / BFS / autotune scripts** — Python drivers (`autotune_sweeper_with_all_ideas.py`, `bfs_sweep.py`, `f16_sweeper_2gpu.py`, etc.) orchestrate compiling variants from `sweeper_srcs` or templates and recording outcomes.

4. **Kernel bank** — Materialized `kernel.cu` per shape for batch compile (`kernel_bank/compile_all.sh`) and verify+measure (`run_all.sh`, `run_f16_bank.sh`), updating `meta.json` and registries.

5. **Push-marginal** — `push_marginal_queue.json` + `push_marginal_sweeper.py` track a queue of shapes that need extra work vs cuSPARSELt.

6. **Paper / analysis** — `analysis_report.py`, `build_readout.py`, `compare_bfs_vs_cusparselt.py` aggregate logs for reporting.

This directory is artifact-heavy (logs, perf, sources) by design so runs are reproducible and resumable after context loss or crashes.
