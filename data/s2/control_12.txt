Task: JSON config files and schema summaries (control_12)

Note: The repository contains hundreds of JSON files (kernel bank `meta.json`, checkpoints, logs, `node_modules`, experiment data). Below are the **main configuration / schema** categories used to operate the project.

## 1. kernels/manifest.json (kernel registry — authoritative)

- **version**, **description**
- **hardware** — GPU model, SM arch, CUDA/driver, peak TFLOPS, SM count, SMEM/L2 sizes
- **choreo_repo**, **gemm_sp_paper_repo**, **gemm_sp_kernel_dir** — absolute paths to external repos
- **reference_kernels** — Per dtype: `best_cu`, `best_tflops`, `tuned_shape`, `origin_iter`, `origin_readme`
- **baseline_kernels** — Filenames of canonical `.co` baselines per dtype
- **scenarios** — Seven scenario definitions (`square`, `sweep_m`, `sweep_n`, `sweep_k`, `sweep_mn`, `sweep_mk`, `sweep_nk`) with `sizes` or `sub_series` for tied dimensions
- **size_constraints** — min M/N/K
- **region_clustering** — Near/far thresholds, `typical_shapes_for_scratch` list
- **build** (if present in full file) — nvcc flags, includes, libs templates

## 2. tuning/state.json (global tuning progress)

- **version**, **description**
- **shapes** — Map keyed by `<dtype>_<M>x<N>x<K>`:
  - **status** (e.g. done)
  - **mode** (from_current_best, etc.)
  - **current_iter**, **best_iter**
  - **best_tflops**, **baseline_tflops**
  - **best_kernel** — Path under `kernels/`
  - **scenarios** — List of scenario keys this shape satisfies
  - **note**, **consecutive_discards** (optional)

## 3. tuning/checkpoints/<shape_key>.json (per-shape session snapshot)

Typically mirrors FSM-oriented fields: iteration, best kernel path, partial metrics — used for resume (exact keys vary by tooling version).

## 4. .claude/skills/fsm-engine/state/*.json (CroqTuner FSM)

Example: **loop-state.json**, **loop-state_from_scratch.json**, **push-marginal-state.json**, **iterative-improve-state.json** (as referenced in skills).

Common structure (illustrative):
- **schema_version**
- **fsm** — `current_state`, `iteration`, `max_iteration`, `shape_key`, `dtype`, `shape` [M,N,K], `mode`
- **guard_flags** — Booleans (gpu checked, ncu ran, compile ok, etc.)
- **metrics** — TFLOPS, decisions, consecutive_discards, bottleneck hints
- **paths** — `shape_dir_logs`, `shape_dir_srcs`, `shape_dir_perf`, `checkpoint_file`, `idea_log`
- **completion_promise**, **last_updated**

## 5. tuning/kernel_bank/<dtype>/<MxNxK>/meta.json (per bank entry)

- **shape**, **M**, **N**, **K**
- **source_type**, **source_tag**, **source_origin**
- **extra_nvcc_flags**
- **prior_tflops**, **cusparselt_tflops**
- **status**, **measured_tflops**, **verified**, **measure_status**
- Optional: **hw_eff_pct** (f16 runs)

## 6. tuning/push_marginal_queue.json

Ordered queue of shapes and metadata for the push-marginal campaign (shape keys, ordering by deficit vs cuSPARSELt).

## 7. figures/package.json (npm)

Standard package manifest: **name**, **version**, **scripts.build**, **dependencies** (d3, jsdom).

## 8. tuning_v3 / aggregates

- **tuning_v3/paper_data.json** — Rows for plotting (e.g. improvement_pct per shape)
- **tuning_v3/all_results.json** — Aggregated experiment results (structure aligns with collection scripts)

## 9. IDE / editor (if present)

- **.vscode/*.json** — Editor settings (not always “config” for builds)

## Summary

The **schemas that matter for automation** are: **manifest.json** (shapes and references), **tuning/state.json** (what is done), **FSM state JSON** (what step the agent is on), and **kernel_bank meta.json** (per-binary provenance and measurements). Everything else is mostly derived logs or paper plot inputs.
