Task 17 — `.claude/skills/` tuning strategies compared

Shared foundation
-----------------
All serious tuning skills integrate the **fsm-engine** (`pre-step-check.sh` → step → `post-step-check.sh` → `state-transition.sh`), guard flags, idea logs, and checkpointing. **Identity** and compaction protocols live under `fsm-engine/protocol/`.

| Skill | Primary goal | Iteration budget | Shape coverage | State file |
|-------|----------------|------------------|----------------|------------|
| **ai-tune-from-current-best** | Adapt reference best `.cu` to each manifest shape (DFS-style deep sweep) | Up to **30** per shape | All **134** shapes/dtype (268 total f16+e4m3), non-stop | `loop-state.json` |
| **ai-tune-bfs** | Breadth-first: baseline + optimizations until **first KEEP** or 30 iters | Up to **30**, early exit on first improvement | Same shape universe, **coverage-first** | `loop-state.json` (INIT with mode BFS semantics) |
| **ai-tune-from-scratch** | Rebuild from baseline `.co` for **far** typical shapes | **≥150** per shape; ncu mandatory each iter | Subset: `typical_shapes_for_scratch` + escalations from from-best | `loop-state_from_scratch.json` (separate to allow parallel sessions) |
| **push-marginal** | Push **18 e4m3** shapes that lose or barely beat cuSPARSELt | **50** choreo-focused iterations + sweeper phase | Fixed queue in `tuning/push_marginal_queue.json` | `push-marginal-state.json`, keys often `*_pm` |
| **iterative-improve** | Improve **verified kernel bank** entries vs cuSPARSELt | Long-running (**999** max in skill), multi-phase | Picks worst speedup shape from `tuning/kernel_bank/e4m3` | `iterative-improve-state.json` |
| **ralph-loop-request** | Meta: continuation gate for agents | N/A | N/A | N/A |

Strategic differences
---------------------
1. **Seed kernel** — From-current-best and BFS start from the **reference adapted** kernel; from-scratch and push-marginal phase-1 often start from **baseline** `.co` for architectural freedom.

2. **Termination** — From-current-best runs **full 30** iters per shape (per workspace rules: no early exit on discards). BFS explicitly **stops at first KEEP** to spread effort across shapes.

3. **Profiling obligation** — From-scratch **requires ncu every iteration**; from-current-best uses ncu when useful; push-marginal emphasizes structural choreo exploration then sweeper.

4. **Artifact keys** — From-scratch uses **`_fs` suffix** on directories to isolate from from-best runs for the same shape.

5. **push-marginal** — Two phases (choreo DSL exploration, then automated sweeper ideas); targets competitive gaps vs cuSPARSELt only.

6. **iterative-improve** — Assumes **bank** already has verified `meta.json`; cycles choreo phases, NCU, sweeper, compare, optional bank update — orthogonal to the manifest-wide sweeps.

Dispatcher
----------
The `.cursor/rules/perf-tune.mdc` router (referenced by skills) chooses **from-current-best vs from-scratch** using shape region and prior results — users invoke `/perf-tune` rather than picking skills manually when following project conventions.
