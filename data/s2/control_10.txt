Task: Analyze tuning/ directory structure and workflow (control_10)

## Top-level layout (tuning/)

- **state.json** — Global registry of per-shape tuning status (done/in-progress), modes (from_current_best / from-scratch), best kernel paths, scenarios covered, notes.
- **logs/** — Per-shape iteration logs: `logs/<dtype>_<M>x<N>x<K>/results.tsv` (and related traces).
- **srcs/** — Kernel source iterations per shape key (`<dtype>_<M>x<N>x<K>` or `_fs` for from-scratch).
- **perf/** — Timing and profiling artifacts per shape (e.g. `timing_*.txt`, ncu outputs), organized by shape key.
- **checkpoints/** — Crash-safe JSON checkpoints per shape (`<key>.json`) for resume.
- **kernel_bank/** — Curated “bank” of materialized CUDA per (dtype, shape): `kernel_bank/<dtype>/<MxNxK>/kernel.cu`, `meta.json`, compiled `kernel` binary; plus aggregate `registry_*.tsv`, `results_*.json`, build/measure scripts.
- **sweeper_srcs/** — Variant `.cu` sources for automated idea sweeps (per-dtype subdirs like `f16/`, `e4m3/`), named by experiment tag (e.g. `ns200`, `l2_lhs_128b`).
- **sweeper_logs/** — Outputs from sweeper runs (e.g. per-dtype `results.json`, per-idea JSON).
- **exhaust_logs/**, **exhaust_srcs/** — Exhaustive sweep experiment logs and sources.
- **combo_logs/**, **combo_srcs/** — Combo / multi-idea exploration artifacts.
- **best_suite/** — Scripts to compile and measure a “best suite” of kernels (`run_best_suite.py`, `resweep_best.py`).
- **push_marginal_logs/** — Logs for push-marginal campaigns (shapes that lose or barely beat cuSPARSELt).
- **f16_remeasure/** — Remeasurement runs for f16.
- **Python tooling** — Root-level drivers: autotune sweeps (`autotune_sweeper_with_all_ideas*.py`), BFS sweep (`bfs_sweep.py`), cuSPARSELt baselines (`cusparselt_*.py`, `cusparselt_baseline.cu`), comparisons, analysis (`analysis_report.py`, `build_readout.py`), 2-GPU sweeper (`f16_sweeper_2gpu.py`), marginal push (`push_marginal_sweeper.py`), tests (`test_sweeper_isolation.py`).

## Workflow represented

1. **Agent-driven deep tuning (primary narrative)** — Shapes are keyed by dtype and M×N×K; artifacts stay isolated per key. Progress rolls forward in iterations: compile → verify → measure → log → checkpoint → (optional) register best under `kernels/`. This aligns with Cursor rules (`perf-tune`, `tuning-workspace.mdc`): near-region shapes adapt from reference best (~30 iters); far-region shapes use from-scratch baseline `.co` (~150 iters, `_fs` suffix).

2. **Sweeper / idea catalog** — A fixed catalog of “ideas” (macros, nanosleep probes, L2 policies, wait depths, etc.) is applied via copies under `sweeper_srcs/`, compiled, and measured; logs land in `sweeper_logs/`. Fast mode can compile once per idea and sweep many shapes via CLI dims.

3. **Kernel bank pipeline** — Selected kernels are materialized into `kernel_bank/<dtype>/<shape>/`, batch-compiled (`kernel_bank/compile_all.sh`), verified and timed (`run_all.sh`, `run_f16_bank.sh`), with `meta.json` holding provenance, TFLOPS, cuSPARSELt comparison, flags.

4. **Exhaust / combo / best_suite / push_marginal** — Specialized campaigns: exhaustive grids, combination exploration, re-validation of best sets, and targeted pushes for underperforming shapes.

5. **Paper-facing aggregation** — `perf/` and TSV/JSON logs feed plots and tables; `build_readout.py` and related scripts consolidate results.

Overall, **tuning/** is an experimentation factory: isolated per-shape workspaces, automated sweep machinery, and a bank layer for verified, comparable kernels versus baselines.
