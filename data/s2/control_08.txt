FSM architecture (from `.claude/skills/fsm-engine/SKILL.md`)

Purpose
- “CroqTuner” wraps AI tuning skills in a finite state machine so protocol steps (build, verify, profile, ideate) are enforced by scripts and persisted state instead of model memory.

Core mechanisms
- Mode-specific JSON state files so different tuning modes can run concurrently in separate sessions (e.g. `state/loop-state.json` for from_current_best, `state/loop-state_from_scratch.json` for from_scratch, `state/iterative-improve-state.json` for iterative_improve). Environment variable `CROQTUNER_STATE_FILE` selects which file all shell hooks read.

Execution loop (every step)
1) Read state JSON first (or initialize if missing via INIT transition).
2) `pre-step-check.sh <STATE>` — blocks illegal transitions.
3) Execute actions from `protocol/step-checklists.md` for the current FSM state.
4) Update `guard_flags` / `metrics` in state with `jq`.
5) `post-step-check.sh <STATE>` — verify mandatory outputs.
6) `state-transition.sh <NEXT_STATE>` — advance FSM; repeat without pausing between steps.

Supporting artifacts
- Append-only idea dedup log (JSONL path in state) to avoid repeating failed ideas.
- `compaction-summary.md` for human-readable resume after context loss; compacted instructions tie back to `fsm.current_state` and iteration.

Mode behavior (summary)
- from-best: max 30 iterations; seed from reference `.cu` in manifest; ncu on a schedule (iter 1, every 5–10, or after discard streak).
- from-scratch: max 150 iterations; seed from baseline `.co`; ncu mandatory before each IDEATE; choreo + nvcc build path.
- iterative-improve: long-running bank optimization with its own state file and state sequence (SELECT_SHAPE → … → UPDATE_BANK → loop).

Sweep policy
- Explicit “non-stop” rule: finishing one shape immediately starts the next using `tuning/state.json` and INIT; anti-stop rules forbid asking the user to continue.

Protocol references (named in skill)
- `protocol/identity.md`, `protocol/loop-contract.md`, `protocol/step-checklists.md`, `protocol/idea-diversity-rules.md`, `kernels/manifest.json`, optional `tuning/state.json`.

In one sentence: the FSM is a file-backed controller with shell pre/post guards, jq-updated state, dedup logs, and compaction notes, layered on top of the existing ai-tune skills.
