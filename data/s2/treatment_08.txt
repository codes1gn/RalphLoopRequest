FSM architecture — CroqTuner (from .claude/skills/fsm-engine/SKILL.md)

Goal: Replace ad-hoc LLM memory with an explicit finite-state machine so GPU kernel tuning follows protocol mechanically (guards, validation, dedup, resume).

Core ideas:
1) State machine — Each tuning session has a current FSM state (e.g. INIT → … → IDEATE → …). Transitions are scripted; illegal transitions are blocked by pre/post checks.

2) Dual (multi) state files — Modes can run concurrently in separate sessions:
   - from_current_best → state/loop-state.json (default)
   - from_scratch → state/loop-state_from_scratch.json (env CROQTUNER_STATE_FILE)
   - iterative_improve → state/iterative-improve-state.json
   All shell scripts honor CROQTUNER_STATE_FILE.

3) Control loop — On every invocation: read state JSON first. If missing → cold start (read protocol/identity.md, loop-contract.md, manifest, tuning/state.json, then INIT). If present → resume from fsm.current_state using step-checklists.md and optional compaction-summary.md and idea dedup log.

4) Per-step pipeline — For each step: pre-step-check.sh → execute checklist actions → set guard_flags/metrics in JSON with jq → post-step-check.sh → state-transition.sh to next state. Repeat without pausing between steps.

5) Supporting artifacts — idea-log.jsonl (dedup), compaction-summary.md (human resume after context loss), protocol/ docs for contract and diversity rules.

6) Mode differences — from-best: seed from reference .cu, nvcc build, ncu cadence, max_iteration 30. from-scratch: seed baseline .co, choreo + nvcc, ncu mandatory before IDEATE, max_iteration 150. iterative-improve: separate state file and shape-cycling workflow (delegated to iterative-improve skill).

7) Sweep philosophy — Document stresses non-stop multi-shape sweeps and anti-stop rules (do not ask to continue after each shape); failure handling logs and continues.

Net: FSM + file-backed state + shell gates = reproducible, resumable tuning loop independent of model context length.
