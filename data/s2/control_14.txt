Task: Python imports and dependency graph (control_14)

## Architecture

The CrokTile paper repository does **not** use an installable Python package layout. Almost all `.py` files are **standalone scripts** run with `python3 path/to/script.py`. They rely on the **standard library** (`json`, `os`, `subprocess`, `pathlib`, `re`, `sys`, `time`, `dataclasses`, `concurrent.futures`, `unittest.mock` in tests) and, in a few places, **third-party** libraries for plotting:

- **tuning_v3/plot_options.py**, **plot_comparison.py**, **collect_data.py** — `matplotlib`, `numpy` (and `json`, `os`).

## Internal (project-to-project) imports

There is **essentially one** explicit internal module dependency:

- **tuning/best_suite/resweep_best.py** → `from run_best_suite import compile_and_measure`  
  (Requires running with `tuning/best_suite/` on `PYTHONPATH` or as `cd tuning/best_suite && python3 resweep_best.py`.)

- **tuning/kernel_bank/build_bank.py** contains duplicated/transformed logic **commented** as derived from `run_best_suite.py` but does not import it as a package.

## Dependency graph (summary)

```
stdlib / subprocess / nvcc / choreo CLI
        ↑
   [every tuning/*.py, tuning_v2/**/*.py, scripts/*.py]

tuning_v3 plotting scripts  →  matplotlib, numpy, json

tuning/best_suite/run_best_suite.py  (leaf script)

tuning/best_suite/resweep_best.py  →  run_best_suite.compile_and_measure
```

## tuning_v2 and tuning trees

No `from tuning_v2...` or `import tuning` edges: each script is a **dag root** invoking the OS and JSON files.

## scripts/

- **generate_shape_suite.py** — `json`, `sys`, `collections.defaultdict`
- **generate_state.py** — `json`

## Conclusion

The “dependency graph” is **flat**: hundreds of edges from scripts to **stdlib** and **external tools** (nvcc, choreo binary via subprocess), not a Python package graph. The only notable internal edge is **resweep_best → run_best_suite**.
